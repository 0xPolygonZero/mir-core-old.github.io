Mir

Introducing Mir

Mir is a protocol for mapping existing identifiers, like social media accounts, email addresses, and names, to sets of public keys, and for recovering from the loss or theft of private keys.

More simply, Mir allows anyone to use cryptographic keys more easily and more securely, for things like authentication, payments, and private communication.

Further, Mir allows users to escape walled gardens, by allowing identifiers to be used independently of their originating services. For example, consider the hypothetical social network: MugTome. Mir would allow users to create and distribute posts to their existing networks, authenticated under their MugTome identifiers, without ever touching MugTome servers.

Mir's goal is an Internet that is more secure, more equitable, and more free.

What is Mir?

Mir is short for Peer-to-Peer Identification and Recovery, and is a protocol that maps existing, human-understandable identifiers, like social media accounts, email addresses, URLs, and names, to addresses that point to sets of public keys.

Users can therefore verify and reason about keys based on the semantic properties of an identifier, rather than a public key fingerprint. For example, a user could send an encrypted message or cryptocurrency directly to an email contact, rather than to a Base 58 encoded string.

Addresses are verifiably recoverable, solving the "lose key material, lose access to everything" problem. Even if all of their devices and backup keys are lost, users can recover their addresses. Further, even if the entire Mir network is compromised by an attacker, addresses are resistant to malicious key bindings.

Mir is maintained by a network of nodes that arrive at consensus about the state of the above mappings. Anyone can run a node in Mir, and Mir's efficient data structures allow for succinct proofs that can be used on mobile and space- and bandwidth-limited devices. Mir is censorship-resistant: anyone can register an address, and anyone can map an identifier to an address.

Some applications of Mir are: improved password-replacement authentication schemes, cryptocurrencies that are usable by everyone, encrypted communication that resists MITM attacks, and alternatives to walled gardens: clients and applications with different incentives, built on the same identifiers.

Why Mir?

Centralized providers control how people transact, authenticate, and communicate on the Internet.

This is bad: single points of failure are vulnerable to breaches; divergent incentives lead to coercion and loss of privacy; and lock-in due to network effects allows providers to force monolithic interfaces and algorithms on all users.

Cryptographic protocols offer an alternative: more security, privacy, and choice, but with reduced usability. Losing key material means the loss of funds and of the ability to authenticate and communicate. Further, key verification is unusable for most users [1,2], and there's no way to verifiably link existing identifiers to keypairs.

A solution to these problems is necessary to reach parity with existing providers. Such a solution should be peer-to-peer: censorship-resistant and designed so that its incentives would necessarily be the incentives of its users.

It should solve the problem of key loss and allow users to resume access, even if all of their devices and backup keys were lost.

It should be capable of uniquely associating existing identifiers with sets of public keys, so that these identifiers could be used to verify public keys and authenticate data.

It should protect the validity of key bindings, even in the case that all other nodes were controlled by an attacker.

Finally, it should support succinct proofs, allowing use on mobile phones and space- and bandwidth-limited devices.

Protocol Overview

Addresses are deterministically created by hashing the combination of an initial set of public keys, and a cryptographic will, an encoding of the conditions under which an address can be updated, recovered, or revoked. Every state of the address is therefore chained to its initial value, and mapping the address to a malicious state without satisfying a cryptographic will at some point in the chain implies finding a collision in the hash function.
