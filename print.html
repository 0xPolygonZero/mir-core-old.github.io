Mir

Introducing Mir

Mir is a protocol for mapping existing identifiers, like social media accounts, email addresses, and names, to sets of public keys, and for recovering from the loss or theft of private keys.

Mir allows everyone to use cryptographic keys more easily and more securely, for things like authentication, payments, and private communication.

Further, Mir enables users to escape walled gardens, by allowing identifiers to be used independently of their originating services. Consider a hypothetical social network: MugTome. Without any cooperation from MugTome, Mir would allow users to distribute posts to their existing networks, authenticated under their MugTome identifiers, without ever touching MugTome servers.

Mir's goal is an Internet that is more secure, more equitable, and more free.

What is Mir?

Mir, short for Peer-to-Peer Identification and Recovery, is a protocol that maps existing, human-understandable identifiers, like social media accounts, email addresses, URLs, and names, to addresses that point to sets of public keys.

Mir allows users to reason about keys based on the semantic properties of an identifier, rather than a public key fingerprint. For example, a user could send an encrypted message or cryptocurrency directly to an email contact, rather than to a Base 58 encoded string.

Addresses are verifiably recoverable, solving the "lose key material, lose access to everything" problem. Even if all of their devices and backup keys are lost, users can recover their addresses. Further, even if the entire Mir network is compromised by an attacker, addresses are resistant to malicious key bindings.

Mir is maintained by a network of nodes that arrive at consensus about the state of the above mappings. Anyone can run a node in Mir, and Mir's efficient data structures allow for succinct proofs that can be used on mobile and space- and bandwidth-limited devices. Mir is censorship-resistant: anyone can register an address, and anyone can map an identifier to an address.

Some applications of Mir are: improved password-replacement authentication schemes, cryptocurrencies that are usable by everyone, encrypted communication that resists MITM attacks, and alternatives clients and applications with different incentives, built on the same identifiers issued by walled gardens.

Why Mir?

A small group of providers controls how Internet users communicate, transact, and authenticate. This is bad: single points of failure are vulnerable to breaches; divergent incentives lead to coercion, addiction, and loss of privacy; and network-effect lock-in allows providers to force monolithic interfaces and algorithms on their users.

Cryptographic protocols offer an alternative to incumbent services, with greater security and privacy, but reduced usability. Key verification and management are difficult [1,2], and there's no way to verifiably link keypairs with existing identifiers. More critically, losing key material means the loss of funds and of the ability to authenticate and communicate.

A solution is necessary to reach parity with existing services. Such a solution should be peer-to-peer, censorship-resistant, and designed so that its incentives would necessarily be the incentives of its users.

It should solve the problem of key loss and allow users to resume access, even if all of their devices and backup keys were lost.

It should be capable of uniquely associating existing identifiers with sets of public keys, so that these identifiers could be used to verify public keys and authenticate data.

It should protect the validity of key bindings, even in the case that all other nodes were controlled by an attacker.

Finally, it should support succinct proofs, allowing use on mobile phones and space- and bandwidth-limited devices.

Architecture

Addresses are deterministically created by hashing the combination of an initial set of public keys, and a cryptographic will, an encoding of the conditions under which an address can be updated, recovered, or revoked. Every state of the address is therefore chained to its initial value, and mapping the address to a malicious state without satisfying a cryptographic will at some point in the chain implies finding a collision in the hash function.
