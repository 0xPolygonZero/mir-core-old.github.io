Mir

Architecture

Mir is designed to maintain two Merkle Binary Prefix Trees: the address tree and the identifier tree, following [1]. In a Merkle Binary Prefix Tree, or Patricia Tree, the path to a leaf is uniquely determined by the prefix of the key (in the key/value sense), such that traversing from the root to a leaf iterates over bits in the prefix and selects the left or right child of a node depending on whether the selected bit is 0 or 1. This yields efficient proofs that a key is either absent from the tree (proof-of-absence), or included in the tree (standard Merkle proof) and included only once, so therefore uniquely bound to a value.

Updates to the address tree and identifier tree (transactions) are submitted by nodes in chronologically-ordered Merkle trees (blocks), such that each block references a parent block and the state of both trees after applying the parent block (or multiple parent blocks, depending on the consensus protocol).

Let \( H:\{0,1\}^* \to \{0,1\}^{256} \) be a cryptographic hash function. Let \(C\) be the hash of the encoding of a cryptographic will. Let \(B\) be the root hash of a previous block, let \(K\) be the Merkle root of a tree consisting of public keys. An address \(A\) and bound value \(V\) are registered as: $$ A = H(K || B || C ), V = A $$ The binding \(V:A\) is added to a block with a valid proof-of-absence of \(A\) in the address tree. This is discussed further in address structure, but this construction guarantees even if all of the nodes in the network are malicious, they can't arbitrarily bind a malicious key to an address and prove to an honest third-party that the binding is valid, since the value of an address at any point is chained back to the original cryptographic will.

This is critical for Mir, since we want to prove that an address or identifier is uniquely mapped to a value - allowing a prover to equivocate, or prove two different values are associated to the same key, would permit a MITM attack.
